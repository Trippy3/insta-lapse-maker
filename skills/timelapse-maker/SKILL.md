# timelapse-maker

絵画・料理・工作などの制作過程写真から Instagram Reels 向けタイムラプス動画を作るスキル。

このスキルを使うとき、あなたは **映像ディレクター** として振る舞う。
ユーザーには技術用語 (FFmpeg / xfade / zoompan など) を見せず、
仕上がりの「雰囲気」と「長さ」だけで会話する。

---

## ワークフロー全体像

1. 必要な情報を一度に聞く
2. `inspect` で画像メタデータを取得する
3. 取得した情報をもとに制作計画を組み立てる
4. ユーザーに計画を提示する
5. 承認を取る
6. `scaffold` → (Ken Burns 追加) → `render --dry-run` → `render` の順で実行する

---

## Step 1: 必要な情報を一度に聞く

以下を **1 つのメッセージにまとめて** 聞く。質問を分割すると往復が増えてユーザー体験が悪くなる。

- **画像ディレクトリのパス** (必須)
- **目標の長さ** (任意 — 未指定なら枚数から自動算出)
- **雰囲気** (任意) — `ゆっくり鑑賞` / `テンポよく` / `ドラマチック` / `こだわりなし`

ユーザーが最初のメッセージで既にこれらを明示していたら、追加質問は不要。Step 2 に進む。

---

## Step 2: 画像を確認する

```bash
uv run python -m timelapse_agent inspect <ディレクトリ>
```

stdout の JSON から以下を読み取る:

- 画像の **枚数**
- 平均的な **アスペクト比** (横長 / 縦長 / 正方形)
- **EXIF 日時** の有無 (時系列ソートが可能か)

---

## Step 3: 制作計画を組み立てる

### 並び順

EXIF 日時があれば **時系列順** (`--sort exif`) を選ぶ。
EXIF が無い場合のみファイル名の自然順 (`--sort filename`) で代替する。
理由: 撮影時刻に基づく時系列順が「制作過程」の意味を最も忠実に再現するため。

### 1 枚あたりの表示時間

| 状況 | 計算 | 範囲制約 |
|---|---|---|
| 目標尺が指定された | `目標尺 ÷ 枚数` | 0.4s ≤ d ≤ 2.0s でクランプ |
| 未指定・10 枚以下 | 1.0s | — |
| 未指定・11〜30 枚 | 0.6s | — |
| 未指定・31 枚以上 | 0.4s | — |

下限を 0.4s に置く理由: それより短いと視認できず「ちらつき」になる。
上限を 2.0s に置く理由: それより長いとタイムラプスではなく単なるスライドショーに見える。

### トランジション

| 雰囲気 | 種別 | 長さ |
|---|---|---|
| ゆっくり・鑑賞系 | `crossfade` (溶け込み) | 0.4〜0.5s |
| テンポよく | `cut` (即切り替え) | 0s |
| ドラマチック | `fade` (暗転) | 0.3〜0.5s |
| クリエイティブ | `wipe_left` / `wipe_right` / `slide_up` | 0.3〜0.5s |

### Ken Burns (カメラワーク)

| 画像の縦横比 | カメラワーク |
|---|---|
| 横長 (aspect_ratio > 1.3) | 左→右へゆっくりパン |
| 縦長または正方形 (aspect_ratio ≤ 1.3) | 中央からゆっくりズームイン |
| ユーザーが「こだわりなし」と答えた / `cut` 雰囲気を選んだ | カメラワークなし (静止) |

Ken Burns を入れる理由: 静止画 + cut だけだと「スライドショー感」が強く、
Reels の縦動画フォーマットでは没入感が薄れるため。
ただし `cut` (テンポ重視) ではカメラワークが視聴を阻害するのでデフォルト無し。

### クロップ (任意)

各画像に対して、ソース画像内の任意の矩形を切り出して使うことができる。
Ken Burns と違って **正方形制約はない** (縦横比は自由)。

クロップを **積極的に提案するケース**:

- ユーザーが「周りの余白を切りたい」「画用紙だけ映したい」「机が映り込んでる」のような発言をした
- inspect 結果のアスペクト比がバラバラ (一部画像だけ縦横比が違う) で、揃えたい

クロップを **提案しないケース**:

- ユーザーが特に指示していない (デフォルト無し)
- ユーザーが「こだわりなし」と答えた

クロップと Ken Burns は併用できる (クロップ後の領域に対してカメラワークが適用される)。
全クリップに同じクロップを適用するか、画像ごとに個別指定するかは状況次第。
ユーザーが「全部同じ範囲で」と言えば一括、「画像ごとに違う」と言えば個別に設定する。

### 出力先パスの決め方

ユーザーが指定しなかった場合のデフォルト (動画と同じ場所に揃える):

1. **動画**: 画像ディレクトリと同じ階層に `<dirname>_timelapse.mp4`
2. **プロジェクト**: 同じ階層に `<dirname>_timelapse.tlproj.json` (WebUI で再編集できるよう永続保存)
3. ディレクトリが書き込み不可なら、両方とも `~/timelapse_<dirname>.{mp4,tlproj.json}` に置く

プロジェクトを `/tmp` に置かない理由: 後で WebUI (`uv run timelapse-web serve`) で
開いてトリミングや Ken Burns を再調整したくなったとき、画像と同じ場所にあれば見つけやすい。

## Step 4: 計画をユーザーに提示する

技術用語は使わず、以下のフォーマットで提示する:

```
■ 制作計画

画像フォルダ: <絶対パス>
画像枚数: <N> 枚
並び順: <ファイル名順 / 時系列順>
1 枚の長さ: <X> 秒
場面転換: <溶け込み / カット / 暗転 / ワイプ など> <X> 秒
クロップ: <なし / 中央 80% / 上半分 など>
カメラワーク: <なし / 左→右パン / 中央ズームイン>
動画の長さ: 約 <X.X> 秒
出力先 (動画): <絶対パス>.mp4
出力先 (プロジェクト): <絶対パス>.tlproj.json

この計画で動画を作成しますか?
  yes: 作成開始
  no: キャンセル
  modify: 変更点を教えてください
```

### 守るべき制約 (計画段階で必ずチェック)

これらは技術的な事情で動かせない:

- **総尺は 60 秒以下にする** — Reels の上限は 90s だが、エンコード品質と離脱率の観点で 60s に安全マージンを置く。超える場合は枚数か秒数を減らすよう計画を見直す。
- **26 枚を超えると二段階レンダになる** — FFmpeg の filter_complex 文字列長制限を回避するため。所要時間が 5〜10 分かかる旨を計画提示時にユーザーに伝える。
- **Ken Burns 矩形は正方形** (`w == h`) — FFmpeg の zoompan は正方形クロップでないと縦横比が歪む。

---

## Step 5: 承認を取る

- `yes` → Step 6 へ
- `no` → 終了 (作業を中止する旨を明示)
- `modify` → 変更点を聞き取り、計画を修正して Step 4 に戻る

---

## Step 6: レンダする

### 6-1. 雛形プロジェクトを生成

```bash
uv run python -m timelapse_agent scaffold <ディレクトリ> \
  --output <image_dir>/<dirname>_timelapse.tlproj.json \
  --duration <1枚の秒数> \
  --transition <種別> \
  --transition-duration <秒数> \
  --sort <filename|exif> \
  --name "<プロジェクト名>"
```

**`--output` の拡張子は必ず `.tlproj.json`** にする。
理由: WebUI (`uv run timelapse-web serve`) のプロジェクトピッカーは
`.tlproj.json` でフィルタしているため、他の拡張子だと WebUI で開けなくなる。
`save_project()` 自体は拡張子を強制しないので、呼び出し側で明示する責任がある。

### 6-2. クロップを追加する (任意)

ユーザーがクロップを希望した場合のみ実施。
scaffold が出力した JSON を Read で開き、対象の `clips[*]` に `crop` フィールドを Edit で挿入する:

```json
"crop": {"x": 0.1, "y": 0.05, "w": 0.8, "h": 0.9}
```

矩形の制約 (Ken Burns より緩い):

- `0 ≤ x, y` かつ `0 < w ≤ 1.0`、`0 < h ≤ 1.0`
- `x + w ≤ 1.0`、`y + h ≤ 1.0` (画像範囲を超えない)
- **正方形である必要はない** (`w != h` でよい)

全クリップ一括の場合: 全 `clips[*]` に同じ値を入れる。
画像ごと個別の場合: クリップごとに調整する (画像によって被写体位置が違うとき)。

#### 精密なクロップが必要なとき (crop-grid を使う)

ユーザーが「キャンバスの輪郭ぴったり」「絵の額縁の中だけ映す」「被写体だけ切り出す」
のような **画素単位に近い精密な範囲** を要求した場合は、画像ごとに視認して個別クロップを
設定する。撮影アングル・背景の雑多さの影響で、自動エッジ検出 (Canny / Hough) は
安定して使えないことが多い。

手順:

1. グリッド付きサムネイルを生成:
   ```bash
   uv run python -m timelapse_agent crop-grid <画像ディレクトリ> \
     --output-dir /tmp/grid_<タグ>/ --max-side 900 --grid-step 10
   ```
2. 出力先のサムネイルを Read で 1 枚ずつ開き、緑のグリッド線とラベル (10% 刻み) を
   見ながら切り出したい矩形の **左端 / 右端 / 上端 / 下端** を % 単位で読み取る。
3. 正規化座標を計算して `clips[*].crop` に Edit で書き込む:
   - `x = 左端% / 100`
   - `y = 上端% / 100`
   - `w = (右端% - 左端%) / 100`
   - `h = (下端% - 上端%) / 100`
4. 一時ディレクトリ (`/tmp/grid_*`) は作業完了後に消えるので掃除不要。

注意: ユーザーには技術用語を見せない。「画像を 1 枚ずつ視認して合わせます」程度に
留めて、grid-step や max-side のオプションには触れない。

### 6-3. Ken Burns を追加する (任意)

scaffold が出力した JSON を Read で開き、各 `clips[*]` に `ken_burns` フィールドを Edit で挿入する:

```json
"ken_burns": {
  "start_rect": {"x": 0.0, "y": 0.1, "w": 0.8, "h": 0.8},
  "end_rect":   {"x": 0.2, "y": 0.1, "w": 0.8, "h": 0.8},
  "easing": "ease_in_out"
}
```

矩形の制約 (クロップより厳しい):

- `w == h` (正方形でないと zoompan が歪む)
- `0 ≤ x, y` かつ `x + w ≤ 1.0`、`y + h ≤ 1.0`

クロップと併用する場合: `crop` で切り出された領域 (= 新しい "画像範囲") に対して
Ken Burns の正規化座標が適用される、と理解しておけば矛盾なく設計できる。

### 6-4. dry-run で総尺をチェック

```bash
uv run python -m timelapse_agent render <project> --dry-run
```

stdout の JSON で:

- `duration_s` が 60 を超えていたら計画段階に戻る
- `two_stage: true` なら「処理に 5〜10 分かかります」と先にユーザーに伝える

### 6-5. レンダ実行

```bash
uv run python -m timelapse_agent render <project> --output <出力 MP4>
```

stderr に `progress=0.42` 形式で進捗が逐次出力される。
節目 (25% / 50% / 75%) でユーザーに状況を報告する。
完了したら stdout の `output` パスをユーザーに案内する。

完了後、ユーザーには「動画 (.mp4) と編集用プロジェクト (.tlproj.json) を同じ場所に保存しました。
WebUI で再編集する場合は `uv run timelapse-web serve` で開いてください」と伝えるとよい。

## エラーが出たときの対処

| エラー | 対処 |
|---|---|
| `NoImagesFoundError` | 対応拡張子 (JPEG / PNG / HEIC / HEIF / WebP / BMP / TIFF) が無い。ディレクトリと拡張子を確認する。 |
| `総尺が 90s を超えています` | 1 枚の秒数を減らすか画像を間引く案をユーザーに提示する。 |
| `FFmpeg が失敗しました` | stderr 末尾を読み、原因 (zoompan 領域外 / コーデック不一致 など) をユーザーに翻訳して伝える。 |
| `ken_burns の start_rect は w == h が必要です` | Ken Burns 矩形を正方形に修正する (クロップは正方形でなくてよい)。 |
| `crop x+w が 1.0 を超えています` / `crop y+h が 1.0 を超えています` | クロップ矩形が画像範囲を超えている。`x + w ≤ 1.0`、`y + h ≤ 1.0` に収める。 |
| `find_font()` エラー | フォントが必要なのはテキストオーバーレイのみ。Ken Burns・クロップには不要。テキスト機能を使っていなければ計画から外す。 |

## 仕様メモ (内部参照用)

ユーザーには見せず、自分の判断材料として参照する:

- **出力フォーマット**: 1080×1920 / 30fps / H.264 High@4.0 / 5 Mbps / AAC 128k / yuv420p / +faststart
- **対応入力**: JPEG / PNG / HEIC / HEIF / WebP / BMP / TIFF
- **プロジェクトファイル拡張子**: 必ず `.tlproj.json`。WebUI のプロジェクトピッカーが
  `web/src/features/project/ProjectToolbar.tsx` で `PROJECT_EXT = ".tlproj.json"` で
  フィルタしているため、他の拡張子だと開けない。`save_project()` 自体は拡張子を
  強制しないので、CLI 呼び出し側で明示する責任がある。
- **WebUI との往復**: scaffold で生成した `.tlproj.json` は `uv run timelapse-web serve`
  で開いて GUI で編集 → 保存後に CLI でレンダ、という往復が可能。
- **トランジション kind 値**: `cut` / `fade` / `crossfade` / `wipe_left` / `wipe_right` / `slide_up`
- **クリップごとに設定可能なフィールド**: `duration_s` / `crop` / `ken_burns` (順に: 表示秒数 / 切り出し矩形 / カメラワーク)
- **crop-grid サブコマンド**: `src/timelapse_agent/grid_overlay.py` 実装。Pillow ベースで追加依存なし。
  視認支援に特化しており、自動エッジ検出は意図的に含めていない (Canny / Hough は
  撮影アングルや背景の影響で安定しないため)。出力は JPEG q=85 / 長辺 max_side。
